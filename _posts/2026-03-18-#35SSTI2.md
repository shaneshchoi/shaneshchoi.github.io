# Server-Side Template Injection - Exploit 접근 방법

Part 1에서는 SSTI가 어떻게 발생하고,  
어떻게 찾는지까지 정리했었다.

이제는 한 단계 더 들어가서  
**찾은 뒤에 실제로 어떻게 공격으로 이어지는지**를 본다.

---

## Documentation

SSTI에서 제일 먼저 보는 건 공식 문서다.

처음엔 귀찮은데,  
이걸 안 보면 오히려 시간이 더 오래 걸린다.

템플릿 엔진마다

- 어떤 문법을 쓰는지  
- 어떤 객체에 접근할 수 있는지  
- 위험한 기능이 뭐가 있는지  

이게 전부 문서에 그대로 나와있다.

링크: [https://hacktricks.wiki/en/pentesting-web/ssti-server-side-template-injection/index.html](https://hacktricks.wiki/en/pentesting-web/ssti-server-side-template-injection/index.html)

---

예를 들어 Python 기반 Mako 템플릿을 보면

~~~html
<%
import os
x=os.popen('id').read()
%>
${x}
~~~

이렇게 템플릿 안에서 바로 코드 실행이 된다.

템플릿이 단순 출력이 아니라  
**코드 실행 환경처럼 동작하는 순간**부터 공격이 시작된다고 보면 된다.

## 문서 볼 때 포인트

문서를 읽을 때는 “기능 설명”보다  
**“이걸로 뭘 할 수 있지?”** 이걸 기준으로 보는게 좋다.

특히 이런 기능은 바로 눈에 들어와야 한다.

- 파일 읽기  
- 시스템 명령 실행  
- 환경 변수 접근  

이런게 보이면 거의 그대로 공격 포인트다.

---

ERB 문서만 봐도 이런게 나온다.

~~~ruby
<%= Dir.entries('/') %>
<%= File.open('/etc/passwd').read %>
~~~

그냥 기능 설명처럼 보이는데,  
공격 입장에서는 바로 파일 읽기다.

---

## 이미 있는 payload 활용

엔진을 알아냈으면 그 다음은 검색이다.

~~~text
jinja2 ssti payload
twig ssti rce
erb ssti exploit
~~~

이런 식으로 찾아보면 이미 정리된 payload가 꽤 많이 나온다.

실제 상황에서는 그대로 쓰기보다는  
대상 환경에 맞게 조금씩 바꿔가면서 맞춘다.

---

## 환경 탐색

여기서부터가 핵심이다.

SSTI는 단순히 실행되는지 확인하는게 아니라  
**어디까지 접근 가능한지 확인하는 과정**이다.

대부분의 템플릿 엔진에는  
전체 객체를 담고 있는 공간이 하나 있다.

~~~text
self / environment / context
~~~

이런 이름으로 많이 나온다.

---

여기서 이런 식으로 접근해볼 수 있다.

~~~text
${T(java.lang.System).getenv()}
~~~

환경 변수나 내부 객체를 확인하면서  
쓸 수 있는 것들을 하나씩 늘려가는 단계다.

---

이 과정에서 하는 건 결국 하나다.

~~~text
내가 사용할 수 있는 객체와 기능을 계속 확인한다
~~~

이게 어느 정도 보이면 공격 방향도 자연스럽게 잡힌다.

---

## 개발자가 넣은 객체

여기서 한 번 더 봐야 할 게 있다.

템플릿 엔진 기본 객체 말고  
**개발자가 따로 넘겨준 객체**다.

이게 더 위험한 경우가 많다.

- 내부 데이터 들고 있거나  
- 특정 기능 실행 가능하거나  
- DB랑 연결되어 있거나  

이런 경우가 많기 때문이다.

---

그리고 이건 사이트마다 다르다.

그래서 같은 SSTI라도  
페이지마다 되는 게 다를 수 있다.

---

## RCE만 보는건 아니다

SSTI 하면 보통 RCE부터 떠올리는데  
그게 안 된다고 끝나는 건 아니다.

템플릿만으로도 할 수 있는게 꽤 많다.

~~~text
파일 읽기
환경 변수 확인
내부 경로 탐색
API 접근
~~~

이 정도만 되어도 이미 민감 정보는 충분히 털린다.

---

## 정리

SSTI exploit 흐름은 아래와 같이 정리할 수 있겠다..

~~~text
1. 코드 실행 확인
2. 템플릿 엔진 파악
3. 문서 + 기존 payload 참고
4. 접근 가능한 범위 확장
~~~
